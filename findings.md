# Repository Review Against TRSAA Spec

Date: 2026-04-14

Method:

- Static code review of the repository against the TRSAA spec
- Executed verification in a local `.venv` using Python `3.11.14` (`>=3.10` as required by the project)
- Test result: `101 passed, 1 failed` via `.venv/bin/pytest -q`

## Readiness Score

**D**

The repository has a credible MVP shape for a task-governed broker, but it does not yet satisfy the core TRSAA thesis: authority is still effectively attached to possession of a capability record identifier, not to a verified task/runtime/action execution context. Several controls described in the README and spec exist only as metadata or helper code and are not enforced on the live execution path.

## Findings

### 1. Critical: execution is not actually bound to the requesting task or runtime

Relevant code:

- `ExecuteRequest` requires `task_id`, but the handler ignores it entirely: `src/jitauth/core/schemas.py:111-117`, `src/jitauth/broker/routes.py:295-308`
- The gateway authorizes execution only by looking up `capability_id`: `src/jitauth/proxy/gateway.py:131-154`
- JWT capability tokens are minted and a verifier exists, but no execution path accepts or verifies them: `src/jitauth/broker/routes.py:258-289`, `src/jitauth/core/tokens.py:31-126`

Impact:

- Any caller that learns a valid `capability_id` can invoke the capability until expiry or revocation.
- There is no proof that the caller is the runtime the capability was minted for.
- There is no proof that the invocation belongs to the `task_id` supplied in the request body.

Why this fails the spec:

- TRSAA requires task-bound authorization, runtime binding, and machine-checkable capability use. The current implementation is closer to a bearer-ID system than a brokered capability system.

### 2. High: scope and reduced-scope approvals are stored, but never enforced

Relevant code:

- Task and capability models store `resource_scope`: `src/jitauth/core/models.py:114-166`
- Approval requests store `reduced_scope`: `src/jitauth/core/schemas.py:131-135`, `src/jitauth/core/models.py:199-210`
- Capability minting copies only the first action's `resource_scope` for a system: `src/jitauth/broker/routes.py:229-246`
- Gateway execution checks only system and action, not scope or arguments: `src/jitauth/proxy/gateway.py:138-190`

Impact:

- Least privilege is not enforced at execution time.
- Human approval cannot actually narrow authority, even though the API suggests it can.
- Multi-action same-system tasks collapse to one scope value, which may not match all approved actions.

Why this fails the spec:

- The spec requires machine-checkable granular capabilities, approval reductions, and proxy-side validation of resource scope.

### 3. High: policy evaluation is performed on aggregate task sets, not governed action envelopes

Relevant code:

- Rules are evaluated once per task: `src/jitauth/policy/engine.py:59-86`
- Rule matching uses set intersection semantics for `system`, `action`, and `action_class`: `src/jitauth/policy/engine.py:101-133`
- Capabilities are then minted by grouping actions per system: `src/jitauth/broker/routes.py:229-246`

Impact:

- A mixed task can satisfy a rule because one sub-action matches, while other sub-actions hitch a ride inside the same approved task.
- This makes safe classification dependent on the whole task being manually pre-split by the caller, which is exactly what the broker is supposed to govern.

Concrete failure mode:

- A future policy like "allow `calendar.create_event`" would match any task containing that action, even if the same task also contains another write on the same risk tier that was never meant to be allowed.

Why this fails the spec:

- TRSAA calls for structured task actions and policy decisions that compute minimum necessary scope per action/tool/resource, not per-task set membership.

### 4. High: the live audit path is not tamper-evident despite the README claim

Relevant code:

- The production routes use `_audit`, which inserts plain `AuditEvent` rows without `prev_event_hash`: `src/jitauth/broker/routes.py:375-384`
- The gateway also inserts `AuditEvent` rows directly without hash chaining: `src/jitauth/proxy/gateway.py:207-222`
- The chain-aware logger exists, but is unused by the request lifecycle: `src/jitauth/audit/logger.py:26-67`
- The chain implementation itself depends on process-global memory `_last_event_hash`: `src/jitauth/audit/logger.py:23,45-65`
- README claims "Events are hash-chained for tamper detection": `README.md:139-147`, `README.md:205-213`

Impact:

- Actual audit events generated by the broker are not hash chained.
- Even if the helper were used everywhere, the chain would break across process restarts or multiple workers.

Why this fails the spec:

- TRSAA requires tamper-evident auditability on the real enforcement path, not just in a helper module and unit tests.

### 5. Medium: capability JWTs can be minted already invalid after the database round-trip

Observed test evidence:

- `tests/test_tokens.py::TestCapabilityEndpointJWT::test_capabilities_include_token` fails after a normal task lifecycle because `verify_capability_token()` rejects a newly minted token as "not yet valid": `tests/test_tokens.py:155-179`

Relevant code:

- Capabilities are committed, refreshed from the DB, and then used to mint JWT claims: `src/jitauth/broker/routes.py:261-289`
- `mint_capability_token()` calls `datetime.timestamp()` directly on the refreshed `issued_at` / `expires_at` values: `src/jitauth/core/tokens.py:31-81`
- The models store timezone-aware datetimes, but SQLite strips timezone metadata on round-trip in this codebase, which is already acknowledged elsewhere in audit hashing: `src/jitauth/core/models.py:156-158`, `src/jitauth/audit/logger.py:103-108`

Impact:

- The "signed capability token" integrity layer is not reliable in the default SQLite-backed setup used by tests and likely local development.
- Freshly minted tokens can be invalid immediately, which undermines the project's claim that capability JWTs are usable proof artifacts.

Likely root cause:

- The refreshed capability datetimes become naive values like `2026-04-14T19:25:40.898018`; `datetime.timestamp()` then interprets them as local time instead of UTC, shifting `iat`/`exp` by the local UTC offset.

### 6. Medium: the SDK decorator bypasses broker-mediated execution

Relevant code:

- The decorator creates a governed task, but then directly executes the wrapped function locally: `src/jitauth/sdk/decorators.py:75-93`

Impact:

- If the wrapped function has its own standing credentials or direct network access, the model/runtime still performs the action outside the execution proxy.
- Audit and capability state exist, but the actual side effect is not enforced by the broker.

Why this fails the spec:

- The spec explicitly separates reasoning and execution and routes risky actions through the broker.

### 7. Medium: secret and data sanitization is incomplete

Relevant code:

- Execution logs sanitize only a small set of input key names: `src/jitauth/proxy/gateway.py:198`, `src/jitauth/proxy/gateway.py:280-290`
- Invocation results are stored verbatim in `result_summary`: `src/jitauth/proxy/gateway.py:201-204`
- HTTP adapter returns response bodies to the runtime without content-level secret scrubbing: `src/jitauth/proxy/adapters/http.py:129-165`
- Shell adapter returns raw stdout/stderr: `src/jitauth/proxy/adapters/shell.py:118-146`

Impact:

- Downstream APIs that echo tokens, PII, or secrets in response bodies will send them straight back to the runtime and store them in audit-related tables.
- The implementation currently protects a few header names and a few obvious input keys, but not the broader "no secret echoing" requirement from the spec.

### 8. Medium: idempotency is cross-task and can return another task's result

Relevant code:

- Idempotency lookup filters only on `idempotency_key`: `src/jitauth/proxy/gateway.py:164-178`
- `ToolInvocation.idempotency_key` has no uniqueness or composite index with `task_id`/`capability_id`: `src/jitauth/core/models.py:181-196`

Impact:

- Reusing an idempotency key across tasks can return a prior task's result.
- This is both a correctness bug and a potential cross-task data leak.

### 9. Medium: the broker startup path does not load adapter configs

Relevant code:

- `jitauth serve` starts the FastAPI app only: `src/jitauth/cli.py:15-29`, `src/jitauth/broker/server.py:19-40`
- Adapter loading is wired only into the MCP server path: `src/jitauth/mcp/server.py:71-74`
- README quick start implies the broker will use `adapters.yaml`: `README.md:17-25`

Impact:

- A user can follow the documented quick start and end up with a broker that cannot execute any configured tools.
- This is a readiness and operability gap, not just a documentation issue.

### 10. Low: audit query API exposes a `runtime_id` filter that is never applied

Relevant code:

- Query parameter declared but unused: `src/jitauth/broker/routes.py:355-369`

Impact:

- Clients cannot actually query audit data by runtime even though the API suggests they can.

### 11. Low: task lifecycle does not reach `completed` or `failed`

Relevant code:

- `TaskStatus` defines `completed` and `failed`: `src/jitauth/core/models.py:32-41`
- The implemented lifecycle stops at `executing` after capability minting: `src/jitauth/broker/routes.py:250`

Impact:

- Task state is not a reliable lifecycle record.
- This weakens audit completeness and operational observability.

## Code Inefficiencies

### 1. `verify_audit_chain` is O(n^2)

Relevant code:

- `events.index(event)` is called inside the loop: `src/jitauth/audit/logger.py:88-95`

Impact:

- Verification work grows quadratically with event count.

### 2. JSON-heavy security fields are stored as opaque `Text`

Relevant code:

- `allowed_actions`, `resource_scope`, `arguments`, `result_summary`, `reduced_scope` are plain text blobs: `src/jitauth/core/models.py:149-152`, `src/jitauth/core/models.py:188-191`, `src/jitauth/core/models.py:206`

Impact:

- The broker repeatedly serializes/deserializes policy-critical fields.
- Database-level validation, indexing, and targeted querying are all much harder than they need to be.

### 3. Idempotency lookups will degrade as invocation volume grows

Relevant code:

- Query scans by `idempotency_key` but no index exists on that column: `src/jitauth/proxy/gateway.py:164-178`, `src/jitauth/core/models.py:190`

Impact:

- This becomes a hot-path table scan under load.

## Test Gaps

The existing tests cover the happy path well, but they do not appear to cover the highest-risk failures above:

- no test proving `/execute` rejects mismatched `task_id`
- no test proving runtime identity or JWT binding is enforced
- no test proving `resource_scope` or `reduced_scope` changes execution
- no test proving route-generated audit events are actually hash chained
- no test proving `jitauth serve` loads adapters
- no test for idempotency-key collisions across tasks

One important exception: the suite does catch a real JWT bug today. The executed run failed on `tests/test_tokens.py::TestCapabilityEndpointJWT::test_capabilities_include_token`, which is now reflected in finding 5 above.

## Bottom Line

This repository is a promising prototype for a brokered auth system, but it is not yet a TRSAA-compliant enforcement boundary. The largest gap is not missing polish; it is missing enforcement of the core security invariants. Based on the executed test run, I would first fix execution binding, scope enforcement, approval reduction handling, the real audit chain, and the JWT timestamp bug, then add tests specifically for those controls.

## Prioritized Fix List

### Priority 0: fix broken correctness on the current path

1. Fix JWT timestamp handling so freshly minted capability tokens verify immediately.
   Target areas: `src/jitauth/broker/routes.py`, `src/jitauth/core/tokens.py`
   Why first: one shipped test already fails here, and it undermines the advertised capability-token model.
   Expected outcome: `.venv/bin/pytest -q` returns fully green on the current suite.

2. Add an execution-time binding check between `ExecuteRequest.task_id` and the loaded capability.
   Target areas: `src/jitauth/broker/routes.py`, `src/jitauth/proxy/gateway.py`
   Why first: this is the simplest high-impact enforcement gap to close immediately.
   Minimum bar: reject execution if `req.task_id != cap.task_id`.

### Priority 1: restore the core TRSAA enforcement boundary

3. Require a verifiable runtime-bound proof on `/execute`, not just `capability_id`.
   Target areas: `src/jitauth/core/schemas.py`, `src/jitauth/broker/routes.py`, `src/jitauth/core/tokens.py`, `src/jitauth/proxy/gateway.py`
   Recommended direction: accept a capability token on execute, verify signature/issuer/expiry, and confirm token claims match the DB capability record.
   Minimum checks:
   - token `sub` matches `capability_id`
   - token task/runtime/system/action claims match the capability row
   - request `task_id` matches both token and DB
   Why now: without this, authority is still effectively attached to a leaked identifier.

4. Enforce `resource_scope` and approval-reduced scope at execution time.
   Target areas: `src/jitauth/broker/routes.py`, `src/jitauth/proxy/gateway.py`, adapter layer
   Recommended direction: represent scope as structured JSON, merge policy scope plus approval reduction into the minted capability, then validate arguments against that scope before dispatch.
   Why now: least privilege is currently metadata only.

5. Refactor policy evaluation to operate on explicit action envelopes, not task-wide set intersections.
   Target areas: `src/jitauth/policy/engine.py`, capability minting flow in `src/jitauth/broker/routes.py`
   Recommended direction: evaluate each `TaskAction` or normalized action envelope independently, then compose a capability bundle from approved actions only.
   Why now: this is required before more granular policies can be trusted.

### Priority 2: make audit and approvals real rather than nominal

6. Replace route-local `_audit()` writes with a single chain-aware audit writer used on the real request path.
   Target areas: `src/jitauth/broker/routes.py`, `src/jitauth/proxy/gateway.py`, `src/jitauth/audit/logger.py`
   Recommended direction: persist the previous hash from the database, not a process-global variable.
   Why now: current audit entries are not tamper-evident despite the product claim.

7. Make approvals materially affect authority.
   Target areas: `src/jitauth/broker/routes.py`, `src/jitauth/core/models.py`, capability minting logic
   Recommended direction: when `reduced_scope` is provided, apply it to the approved capability payload and audit exactly what was reduced.
   Why now: approval UX is misleading if reduced scope has no execution effect.

8. Complete task lifecycle state transitions and audit them.
   Target areas: `src/jitauth/proxy/gateway.py`, task orchestration flow
   Minimum bar:
   - move task to `completed` after successful bounded execution
   - move task to `failed` on terminal execution failure
   - record corresponding audit events
   Why now: this improves both governance visibility and operational debugging.

### Priority 3: close data leakage and cross-task correctness bugs

9. Scope idempotency keys to task and capability.
   Target areas: `src/jitauth/proxy/gateway.py`, `src/jitauth/core/models.py`
   Recommended direction: query on `(task_id, capability_id, idempotency_key)` and add a DB index or uniqueness constraint.
   Why now: current behavior can leak another task's prior result.

10. Harden result and log sanitization.
    Target areas: `src/jitauth/proxy/gateway.py`, `src/jitauth/proxy/adapters/http.py`, `src/jitauth/proxy/adapters/shell.py`
    Recommended direction:
    - sanitize output bodies as well as input keys
    - add configurable redaction rules per adapter/system
    - avoid storing raw stdout or full downstream bodies by default for sensitive systems
    Why now: current behavior conflicts with the spec's "no secret echoing" requirement.

11. Remove the local-execution escape hatch from `jitauth_tool`.
    Target areas: `src/jitauth/sdk/decorators.py`
    Recommended direction: either execute through the broker or rename/document this path as non-brokered instrumentation, not governed execution.
    Why now: the current decorator violates the spec's separation of reasoning and execution.

### Priority 4: operability and product honesty

12. Load adapter configuration on broker startup or explicitly remove that claim from the quick start.
    Target areas: `src/jitauth/cli.py`, `src/jitauth/broker/server.py`, `README.md`
    Why now: the current setup path does not match the documentation.

13. Implement the `runtime_id` audit filter or remove it from the API surface.
    Target areas: `src/jitauth/broker/routes.py`, `openapi.json`

14. Replace opaque text blobs for policy-critical JSON fields with structured columns or disciplined typed wrappers.
    Target areas: `src/jitauth/core/models.py`
    Why later: this is not the first security fix, but it will simplify correct scope enforcement and querying.

15. Reduce obvious inefficiencies once the enforcement model is stable.
    Target areas: `src/jitauth/audit/logger.py`, invocation indexing
    Minimum bar:
    - remove `events.index(event)` from chain verification
    - add index support for idempotency lookups
